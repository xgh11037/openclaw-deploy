# OpenClaw 一键部署桌面版使用说明

这份文档对应当前桌面版界面，按现在的软件真实布局来写，目标是让第一次接触 OpenClaw 的用户也能顺着界面跑通。

当前主入口分为 4 个：

- `首页`
- `聊天`
- `调教中心`
- `修复中心`

其中 `AI 服务配置` 是首页流程中的关键一步，不需要手改配置文件。

## 1. 这款软件是做什么的

它是一个面向小白的 `OpenClaw 图形化控制台`，核心作用只有四件事：

1. 检查环境并安装 OpenClaw。
2. 配置 AI 服务和模型，不用自己写 `.env`。
3. 创建 Agent，并把 Telegram、QQ、飞书等渠道接到 Agent。
4. 提供本地聊天、外部渠道对话和修复排错入口。

如果你第一次接触这些概念，可以先这样理解：

- `Agent`：不同助手身份，比如 `main`、`code-agent`
- `渠道`：Telegram、QQ、飞书、Discord、钉钉这些消息入口
- `网关`：让某个 Agent 真正开始对外收发消息的运行实例
- `Skills`：给 Agent 增加额外能力的扩展

## 2. 最快跑通路线

第一次安装，推荐严格按下面顺序走：

1. 打开 `首页`
2. 完成环境检测
3. 安装 OpenClaw
4. 打开 `AI 服务配置`
5. 选择服务渠道、模型并填入 API Key
6. 点击 `验证密钥`
7. 点击 `保存并启用服务`
8. 进入 `聊天`
9. 先和默认 Agent 发出第一条消息
10. 本地聊天确认没问题后，再去接 Telegram、QQ、飞书

如果你现在只想先让软件能回答问题，做到第 9 步就够了。

安装 OpenClaw 时，程序会自动按顺序尝试：

1. 默认 npm 源
2. `npmmirror`
3. 腾讯云 npm 镜像

如果这三个源都失败，通常不是界面按钮问题，而是当前网络仍然无法访问依赖源或其上游 GitHub 资源。这种情况下在国内网络环境下仍可能需要代理、VPN 或加速器。

## 3. 页面结构说明

### 3.1 首页

首页是总入口，主要负责 3 件事：

- 告诉你环境是否完整
- 告诉你 AI 服务是否配置好
- 告诉你下一步该去哪里

首页常见动作：

- 看环境检测结果
- 继续安装 OpenClaw
- 进入 `AI 服务配置`
- 进入 `聊天`
- 遇到问题时去 `修复中心`

如果你第一次使用，优先看左侧的 `首次跑通路径` 和 `云睿中转站 / API Key`。

### 3.2 AI 服务配置

这是整个软件最常用的配置页之一，也是后续接入和切换 API 服务的主要入口。

当前页面设计原则是：

- 先选服务渠道
- 再选预设模型
- 最后填 API Key

推荐操作顺序：

1. 选择服务商
2. 选择一个预设模型
3. 粘贴 API Key
4. 点击 `验证密钥`
5. 点击 `保存并启用服务`
6. 点击 `立即开始试聊`

什么时候算配置成功：

- 页面显示已接入或已保存成功
- 可以直接进入聊天页
- 聊天页能正常收到回复

如果你只是想先用起来，不要一上来折腾高级参数。

### 3.3 聊天

聊天页是当前主对话页。

页面结构：

- 左侧：Agent 列表和会话摘要
- 右侧：当前 Agent 的聊天内容
- 输入区：直接发送消息

聊天页适合做这些事：

- 验证 AI 服务是否已经接通
- 验证不同 Agent 的风格和用途
- 在多个 Agent 之间来回切换
- 查看本地缓存的聊天记录

聊天页当前和网关相关的直接操作，暂时只保留：

- `启动当前 Agent 网关`

聊天页现在不再提供：

- `一键启动所有 Gateway`
- `一键关闭所有 Gateway`

如果你要做批量启动、批量检查或导出多网关报告，请到 `调教中心` 里的对应区域操作。

当前聊天页有几个重要特点：

- 历史记录优先走本地缓存，避免频繁拉远端造成卡顿
- 切换 Agent 时优先本地切换，网关在线时再同步
- 聊天列表会显示最后一条摘要和时间
- 未读红点会按本地与远端同步结果更新

如果聊天时出现轻微卡顿，优先区分这几类情况：

- 正在流式回复中
- 某个 Agent 当前正在发送中
- 当前会话历史过长
- 网关在线同步刚好发生

### 3.4 调教中心

调教中心是日常配置区，目前按小白能看懂的方式拆成几个入口：

- `Agent 管理`
- `渠道配置`
- `Skills`
- `记忆`
- `高级设置`

你可以这样理解：

- `Agent 管理`：管理助手本身
- `渠道配置`：把 Telegram、QQ、飞书这些入口接进来
- `Skills`：扩展能力
- `记忆`：长期信息
- `高级设置`：模型策略和维护工具

### 3.5 修复中心

修复中心是统一排错页。

适合处理的问题：

- Node、Git、OpenClaw 没装好
- 网关启动失败
- 外部渠道不通
- Skills 缺依赖
- 运行状态异常

当前修复中心会把一些自动状态查询放到后台执行，不会在你一打开页面时把界面卡住。

## 4. 调教中心详细说明

### 4.1 Agent 管理

默认页只保留最核心的 5 个动作：

- 查看 Agent 列表
- 改名
- 设默认
- 新建
- 删除

推荐做法：

1. 先保留 `main`
2. 再增加 1 到 2 个真正会用到的 Agent
3. 名字尽量一眼能懂

例如：

- `通用助手`
- `代码助手`
- `写作助手`

如果你只是小白用户，不建议一开始就创建太多 Agent。

### 4.2 渠道配置

这是把外部渠道接到 Agent 的主页面。

页面布局：

- 左侧：渠道标签
- 右侧：当前渠道配置表单
- 底部固定操作条：常用操作

#### 4.2.1 左侧渠道标签

左侧会显示每个渠道当前状态，帮助你快速判断：

- `待补全`：凭据还没填完整
- `已配置`：已经填好，但还没发布到网关
- `已连通`：当前网关已经接入或检测通过

#### 4.2.2 右侧表单

右侧只负责填写当前渠道需要的凭据。

你不需要懂底层 JSON，只要按界面填：

1. 先选择渠道
2. 点击 `按 Agent 自动生成`
3. 为每个 Agent 填对应凭据
4. 看底部固定操作条
5. 点击 `保存配置`
6. 点击 `测试连通`
7. 点击 `启动当前 Agent 网关`

当前表单有两个很重要的规则：

- 输入框会自动暂存草稿，减少重复丢失
- `保存配置` 会保存当前渠道下你已经改过的凭据，不是只保存当前选中的某一个 Agent 行

#### 4.2.3 底部固定操作条

底部固定条现在只保留最常用的 3 个动作：

- `保存配置`
- `测试连通`
- `启动当前 Agent 网关`

同时会显示：

- 当前 Agent
- 当前渠道
- 当前网关
- 当前结果

这样你不需要来回滚动页面找按钮。

### 4.3 首次配对审批

Telegram、QQ、飞书这类渠道，第一次可能需要配对或审批。

如果外部渠道里返回配对码：

1. 回到 `调教中心 -> 渠道配置`
2. 切到对应渠道
3. 在 `首次配对审批` 区域粘贴配对码
4. 点击 `批准配对码`

当前页面也会后台轮询待审批请求，减少你手动一直点“查询待审批”的需要。

### 4.4 Skills

Skills 是 Agent 的能力扩展。

建议顺序：

1. 先确认聊天正常
2. 再安装 Skills
3. 安装后再做一次修复或检测
4. 仍异常时再去修复中心

如果状态显示：

- `可用`：可以直接使用
- `缺依赖（可修复）`：可以尝试自动修复
- `仍缺依赖（已尝试修复）`：自动修过一次，但问题还没完全解决
- `需手动处理`：需要按提示手动装环境或工具

### 4.5 记忆

记忆用于保存长期信息和偏好。

如果你希望某个 Agent 记住固定背景、工作习惯、口吻偏好，可以在这里配置。

### 4.6 高级设置

高级设置已拆成两组：

- `模型策略`
- `维护工具`

如果你不是进阶用户，建议默认不要展开。

## 5. 常见渠道配置方法

### 5.1 Telegram

推荐流程：

1. 进入 `调教中心 -> 渠道配置`
2. 选择 `Telegram`
3. 点击 `按 Agent 自动生成`
4. 填写每个 Agent 的 `Bot Token`
5. 需要时补充 `Chat ID`
6. 点击 `保存配置`
7. 点击 `测试连通`
8. 点击 `启动当前 Agent 网关`
9. 去 Telegram 给机器人发消息

注意：

- 一个 Telegram Token 不要被多个有效网关同时轮询
- 如果出现 `getUpdates` 冲突，通常就是重复占用了同一个 Token

### 5.2 QQ

当前 QQ 页面使用的是：

- `AppID`
- `AppSecret`

界面会自动按 OpenClaw 需要的结构写入，不需要你再手动拼接命令。

推荐流程：

1. 选择 `QQ`
2. 点击 `按 Agent 自动生成`
3. 填 `AppID`
4. 填 `AppSecret`
5. 点击 `保存配置`
6. 点击 `测试连通`
7. 点击 `启动当前 Agent 网关`
8. 如果出现配对码，再到审批区处理

### 5.3 飞书

飞书最常见的问题是回调地址和首次配对。

推荐流程：

1. 填好应用凭据
2. 确认飞书开放平台中的回调地址配置无误
3. 如果出现配对码，在页面里完成审批
4. 点击 `保存配置`
5. 启动当前 Agent 网关
6. 回到飞书测试

如果提示 `access not configured`，通常不是模型有问题，而是：

- 首次配对没完成
- 访问权限还没放行
- 飞书侧的回调或订阅配置没生效

## 6. 修复中心怎么用

修复中心推荐按这个顺序用：

1. 先点 `一键体检`
2. 看哪一项先报错
3. 能自动修就先自动修
4. 修完再回到聊天或渠道页测试
5. 仍有问题再看运行状态与任务队列

当前修复中心主要看这些：

- 模型探活
- Skills 可用率
- 自检状态
- 记忆状态
- 最近启动输出
- 任务队列

为了减少假死，修复中心中的一部分自动检查已经改为后台任务，页面只负责接收结果。

## 7. 常见问题

### 7.1 环境检测一直失败

优先检查：

1. Node.js 是否安装
2. Git 是否安装
3. 安装后是否重开软件
4. 是否被安全软件拦截

如果环境检测都正常，但安装 OpenClaw 仍然失败，再额外检查：

1. 程序自动切换镜像源后是否仍失败
2. 当前网络是否能访问 npm
3. 是否被公司网络、校园网或安全软件限制
4. 是否需要代理、VPN 或加速器访问上游 GitHub 资源

### 7.2 保存 AI 配置后还是不能聊天

优先检查：

1. API Key 是否验证通过
2. 模型是否已选中
3. 是否点击了 `保存并启用服务`
4. 聊天页能否正常进入

### 7.3 本地聊天能回复，外部渠道不回复

通常按这个顺序排查：

1. 渠道凭据是否填对
2. 是否点击过 `保存配置`
3. 是否点击过 `测试连通`
4. 当前 Agent 网关是否已启动
5. 是否存在首次配对未审批

### 7.4 出现配对码或 `access not configured`

不要回 AI 服务配置页改参数。

正确做法：

1. 去 `调教中心 -> 渠道配置`
2. 切到对应渠道
3. 在 `首次配对审批` 区域处理

### 7.5 浏览器打开对话页失败或拒绝访问

先检查：

1. 网关是否已经启动
2. 是否刚启动还没完全就绪
3. 本机安全软件是否拦截
4. 是否打开了错误的旧地址

### 7.6 Skills 修复后状态还是不对

如果状态是 `仍缺依赖（已尝试修复）`，意思是：

- 程序已经尝试自动修复
- 但缺失依赖还没完全补齐
- 需要继续看具体报错，或者手动安装相关依赖

### 7.7 不知道自定义配置路径填什么

这里填的是 `OpenClaw 配置目录`，不是程序安装目录。

如果你不确定：

1. 优先留空使用默认路径
2. 或使用软件的自动检测结果

## 8. 推荐给新手的默认玩法

### 8.1 单 Agent 本地聊天

适合第一次使用的人。

只做两件事：

- 配好 AI 服务
- 用默认 `main` Agent 聊天

### 8.2 两个 Agent 分工

适合已经跑通基础聊天的人。

例如：

- `main`：通用聊天
- `code-agent`：代码和技术问答

### 8.3 单 Agent 多渠道

适合希望一个助手同时接多个入口的人。

例如同一个 Agent 同时接：

- Telegram
- QQ
- 飞书

这也是当前最推荐的小白外部接入玩法。

## 9. 维护者打包发布

如果你是项目维护者，准备把软件打包上传到 GitHub：

1. 先确认 `npm run build` 和 `cargo check` 都通过
2. 运行 `npm run release`
3. 打包脚本会生成安装包、绿色版压缩包和 `release/` 目录
4. 把生成的安装包和 zip 上传到 GitHub Release

当前发布脚本会显式复制根目录的 `使用文档.md` 到发布目录，避免把其他 Markdown 误带进去。

当前推荐给普通用户的主下载文件是：

- 安装版 `.exe`
- Windows 压缩包 `.zip`

## 10. 最后建议

第一次使用，最值得记住的三条：

1. 先本地聊天跑通，再接外部渠道。
2. 一个 Agent 可以接多个渠道，但网关和凭据一定要按当前 Agent 关系保存清楚。
3. 出问题先去 `修复中心`，不要一上来手改底层配置。
